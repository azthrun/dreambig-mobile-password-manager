## Phase-by-phase detail (0–4, all approved)

**Phase 0 — Scaffolding**: Flutter app `password_manager` (org `com.dreambig`), layered `lib/data/` `lib/domain/` `lib/presentation/`, `go_router`, `flutter_riverpod` DI, `flutter_lints`, l10n scaffolding (`lib/l10n/app_en.arb`), abstract `ApiClient` + `FakeApiClient`. One bug fixed in review (inconsistent `userId` threading across sign-up/sign-in/refresh).

**Phase 1 — Auth & session**: Two-key derivation in `lib/domain/crypto/master_key_deriver.dart` — Argon2id stretches the master secret into IKM, then HKDF-SHA256 with domain-separated `info` labels (`auth-key/v1`, `vault-key/v1`) derives the auth key (only thing ever sent to `ApiClient`) and the vault key (never transmitted). Sign-up flow shows the recovery-mode choice screen (local-only vs remote-backup, side-by-side, consequence text matching GOALS_v2 §1.3) before account creation. Secure storage via `flutter_secure_storage`. Auto-lock (inactivity + backgrounding) enforced via router redirect, not just UI dimming. Biometric unlock (`local_auth`) only gates access to the already-stored vault key, never re-derives it. No issues found in review.

**Phase 2 — Vault core**: Credential item model with a `type` discriminator (`lib/domain/models/vault_item.dart`, `credential_data.dart`) for future non-credential item types without a breaking migration. Local persistence: one JSON file per `userId` (`lib/data/storage/vault_local_store.dart`), each item's secret payload AES-256-GCM-encrypted (`lib/domain/crypto/vault_item_cipher.dart`) with a fresh nonce per encryption, keyed by `AuthState.vaultKey`. Own-account scoping is structural (repository fixed to one userId/vaultKey at construction). 30-day soft-delete/trash (`isPurgeable` uses strict `>`, boundary-tested). Revision history accumulates on edit, restorable. No issues found in review (reviewer added one boundary test as hardening).

**Phase 3 — Password generator**: Charset-based generator (`lib/domain/generator/password_generator.dart`) using `Random.secure()`, configurable length/charsets/ambiguous-char exclusion. Passphrase generator using the **real EFF long Diceware wordlist** (verified byte-for-byte against EFF's published list, all 7,776 words, exact match) — attribution added at `assets/wordlists/NOTICE.md` (CC BY 3.0). Strength estimator is an explicitly-labeled heuristic (not a zxcvbn port — UI says "Estimate only"), wired live into both the generator screen and the vault item form's manual-entry field. No issues found in review.

**Phase 4 — Device registration & encryption**: Per-device X25519 keypair (`lib/data/crypto/device_identity_service.dart`, reusing the existing `cryptography` package). Private key persisted in a dedicated `DeviceKeyStore`. Device state machine in `FakeApiClient`: pending/active/revoked; first device on an account auto-activates, later devices need an already-active session to `approveDevice`. Device management screen with approve/revoke/"simulate new device" (documented single-device-testing stand-in). **Two bugs found and fixed in review**:
1. Sign-out was silently wiping the device private key (shared unnamespaced `flutter_secure_storage` instance meant `SecureStorageService.clear()` nuked device identity too) — fixed by giving `DeviceKeyStore` its own storage namespace.
2. `approveDevice` had no status guard, so approving a `revoked` device silently re-activated it — fixed by requiring current status be `pending`.

Phase 4's reviewer explicitly flagged and deferred to Phase 5: **access tokens are not scoped to device identity** — a session from a pending/revoked device isn't blocked from privileged calls, and revoking a device doesn't invalidate its outstanding tokens. This is Phase 5's core task.

## Phase 6 — Security UX (approved)

Built per GOALS_v2 §2.3-2.5, 2.8:
- **Clipboard hygiene**: `lib/data/clipboard/clipboard_service.dart` — copying a secret (vault item detail, generator screen) starts a 45s auto-clear timer; clearing re-checks the clipboard still holds what was copied before wiping it (won't clobber something else the user copied since). Copying a second secret cancels/replaces the pending timer.
- **FLAG_SECURE (Android screenshot/recording prevention)**: real platform channel (`lib/data/security/secure_screen_service.dart` + `android/app/src/main/kotlin/.../MainActivity.kt`), toggled app-wide on unlock/lock rather than per-screen (documented simplification — most unlocked screens are vault-adjacent).
- **Auto-lock hardening**: `lib/presentation/app/auto_lock_wrapper.dart` now also clears the clipboard immediately on lock, and tracks app lifecycle state so FLAG_SECURE is only cleared once truly resumed-and-locked (not immediately on backgrounding) — this closes a real race where the last unlocked frame could otherwise leak into the Android Recents/app-switcher thumbnail before the OS finishes its snapshot.
- **Secret redaction**: added redacting `toString()` overrides to every secret-bearing domain/data class (`CredentialData`, `AuthSession`, `AuthState`, `VaultItem`, `VaultItemRevision`, and — caught by review — `DerivedKeyMaterial`, `StoredSession`, `StoredDeviceIdentity`) so a stray log/crash-report interpolation can't leak a secret even as a bug.

**Two real bugs found and fixed during review** (the FLAG_SECURE backgrounding race, and the three missed redaction classes) — the phase-reviewer additionally added and validated a regression test for the FLAG_SECURE fix by reverting the fix, confirming the new test failed, then restoring it.

Final state: 103/103 tests passing, `flutter analyze` clean (only pre-existing info-level lints). `IMPLEMENTATION_PLAN.md` status table shows Phase 6 as `done | done | done | Approved — Phase 7 unlocked`.

## Phase 7 — Account deletion & CSV export (approved)

Built per GOALS_v2 §1.7, §3.3:
- **Account deletion**: `lib/presentation/account/delete_account_screen.dart` requires all three — acknowledgement checkbox, master-secret re-auth (via new `AuthController.verifyMasterSecret()`, a genuine constant-time check against the stored vault-key verifier), and typing the literal `DELETE` — before the submit button even enables; no keyboard/enter-key bypass. `AuthController.deleteAccount()` awaits `ApiClient.deleteAccount()` first and only wipes local state (vault store, device identity, secure storage) after that succeeds, so a failed remote call never leaves a false "deleted" local state. `FakeApiClient.deleteAccount` does genuine removal (drops devices, blacklists all outstanding tokens) not flag-setting.

## Phase 8 — Autofill integration (approved)

Built per GOALS_v2 §1.8:
- **Dart side** (fully unit-tested): `lib/domain/autofill/autofill_matcher.dart` matches host/package against vault items, routing exclusively through `VaultRepository.listActive()` for structural own-items-only scoping. `lib/data/autofill/autofill_bridge_service.dart` is a MethodChannel handler wired into `AutoLockWrapper` so it's only registered during an unlocked session — never active while locked or signed out.
- **Native Android** (Kotlin, **not compiled/run** — no Android SDK/emulator available in any sandbox this session had access to; written carefully against the real Autofill Framework API and statically reviewed line-by-line instead): `PasswordManagerAutofillService.kt`, `AutofillStructureParser.kt`, `AutofillBridge.kt`, manifest `<service>` entry + `res/xml/autofill_service_configuration.xml`. Architecture: the vault key never leaves the Dart isolate; the native service either relays through the live, unlocked Flutter engine for real suggestions, or falls back to a static "Unlock to autofill" dataset that launches the app via `PendingIntent` — no decryption ever happens natively, and the fallback UI shows no account-specific info pre-auth.
- **One real bug found and fixed during review**: the original Kotlin wiring filled password fields with the identifier value instead of the secret (the `secret` field was never parsed out of the channel payload) — would have made autofill non-functional/wrong on a real device. Fixed by splitting identifier/password field handling end-to-end and threading `secret` through.
- **Explicitly flagged as needing real-device verification before shipping** (none of this could be exercised in any sandbox this session had access to): `AssistStructure` traversal behavior across real third-party apps/browsers, the `FlutterEngineCache` timeout race, and actual Settings enrollment as the system autofill provider. This was judged an acceptable bar for approval given the infrastructure constraint was known going in, not something any phase could route around.
- **Minor gap noted for Phase 9**: native-side hardcoded strings (e.g. "Unlock to autofill" `RemoteViews` text) are not routed through Flutter's l10n system — flag this explicitly when Phase 9 does its "no hardcoded strings" audit.
- No new Dart dependencies, no regressions. Final: 147/147 tests, clean analyze.
